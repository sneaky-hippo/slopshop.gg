01

Authentication & Access Control

All API calls require a bearer token. Passwords are never stored in plaintext. The signup and login endpoints are independently rate-limited per IP address.

PBKDF2 password hashing — 100,000 iterations
User passwords are hashed with crypto.pbkdf2Sync using SHA-512, 100,000 iterations, and a 16-byte random salt generated with crypto.randomBytes(16). The salt is stored alongside the hash; the plaintext password is never persisted.
auth.js: crypto.pbkdf2Sync(password, salt, 100000, 64, 'sha512')
Bearer token authentication on all API endpoints
Every route under /v1/ (except health, tools, and auth endpoints) passes through a middleware function that validates the Authorization: Bearer <key> header against the in-memory key store backed by SQLite. An invalid or missing key returns HTTP 401 immediately.
server-v2.js: function auth(req, res, next) — applied to all protected routes
Rate limiting on signup: 5 requests / hour / IP
Account creation is limited to five attempts per IP address per rolling hour. Exceeding this limit returns HTTP 429. The counter resets after the window expires and is cleaned up from memory every five minutes.
auth.js: rateLimit('signup:' + ip, 5, 3600000)
Rate limiting on login: 10 requests / minute / IP
Login attempts are limited to ten per IP address per minute to prevent brute-force credential attacks. The response does not differentiate between unknown email and wrong password, returning a generic invalid_credentials code in both cases.
auth.js: rateLimit('login:' + ip, 10, 60000)
API key rotation
Users can rotate their API key at any time via POST /v1/auth/rotate-key. The old key is immediately invalidated and the full credit balance transfers to the new key. Both the in-memory cache and the SQLite database are updated atomically.
auth.js: /v1/auth/rotate-key — old key deleted from apiKeys Map and DB

02

Data Protection

Data in transit is encrypted by the hosting platform. Persistent storage uses SQLite with WAL journaling. Security headers are applied at the application layer via Helmet.js.

HTTPS / TLS enforced in transit
All traffic to slopshop.gg is served over TLS. The API is proxied through Vercel at the root domain. Railway Pro enforces HTTPS and terminates TLS before traffic reaches the application. The server is configured to trust the Railway proxy (app.set('trust proxy', 1)) so that the real client IP is correctly resolved for rate limiting.
server-v2.js: app.set('trust proxy', 1) — Railway handles TLS termination
SQLite with WAL mode for data integrity
The database is opened with journal_mode = WAL (Write-Ahead Logging), which provides crash recovery and allows concurrent reads. All queries use the better-sqlite3 prepared-statement API — no raw string interpolation into SQL.
server-v2.js: db.pragma('journal_mode = WAL')
No plaintext password storage
The users table stores only the PBKDF2-derived hash and the random salt. The plaintext password is never written to disk, logged, or returned in any API response.
users table schema: password_hash TEXT, salt TEXT — no password column
Helmet.js security headers
Helmet is applied as the first Express middleware, setting X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, X-DNS-Prefetch-Control, and other default protective headers on every response. Content-Security-Policy is disabled at the application level because Railway handles it at the edge; Cross-Origin-Embedder-Policy is also disabled for API compatibility.
server-v2.js: app.use(helmet({ contentSecurityPolicy: false, crossOriginEmbedderPolicy: false }))
Encryption at rest
The SQLite database file is not encrypted at rest. If the host filesystem were compromised, the database could be read directly. This is a known gap. See Security Roadmap for more detail.

Sandboxed Code Execution

The exec-javascript API allows users to run arbitrary JavaScript. This runs inside a Node.js VM context with an explicitly restricted global scope — not in the host process context.

Isolated vm.createContext — no fs, net, or process access
User code runs in a context created by vm.createContext(). The only globals exposed are safe, pure-computation primitives: Math, JSON, Array, Object, String, Number, Boolean, Date, RegExp, Error, and a sandboxed console. The Node.js require, process, fs, and net modules are explicitly absent from the context object.
handlers/generate.js: vm.createContext({ console, Math, JSON, Array, Object, ... }) — no require, no process
Execution timeout enforcement
All user-submitted code is executed with a hard timeout. The caller may request a timeout up to 5,000 ms; the default is 1,000 ms. If the code exceeds the limit, the VM is interrupted and an error is returned. Infinite loops cannot run indefinitely.
handlers/generate.js: timeout = Math.min(parseInt(input.timeout, 10) || 1000, 5000)
No network escape from sandbox
fetch, XMLHttpRequest, and Node's http / https modules are not present in the sandbox context. User code cannot initiate outbound network connections or read from the filesystem.
handlers/generate.js — sandbox context (actual code)
const context = vm.createContext({
  console: {
    log:   (...args) => logs.push(args.map(a => typeof a === 'object' ? JSON.stringify(a) : String(a)).join(' ')),
    error: (...args) => logs.push('[error] ' + args.join(' ')),
    warn:  (...args) => logs.push('[warn] '  + args.join(' ')),
  },
  Math, JSON, Array, Object, String,
  Number, Boolean, Date, RegExp, Error,
  parseInt, parseFloat, isNaN, isFinite,
  encodeURIComponent,
  // — require, process, fs, net: not present —
});

04

Input Validation & Safety

Inputs are bounded by size and validated against schemas before reaching any handler. Database queries use parameterized statements exclusively.

1 MB request body limit
Express is configured to reject JSON bodies larger than 1 MB. This prevents memory exhaustion from oversized payloads before any application logic runs.
server-v2.js: app.use(express.json({ limit: '1mb' }))
JSON schema validation on API inputs
Every API has a schema defined in schemas.js. The schema describes required fields, types, and descriptions. Inputs are validated against the schema before being forwarded to the handler. Unknown or malformed fields are rejected with a structured error response.
schemas.js: SCHEMAS[slug].input — per-field type and required definitions
SQL injection prevention via parameterized queries
All database access uses better-sqlite3's db.prepare() with positional parameters. User-supplied values are bound at execution time — they are never interpolated into SQL strings. This eliminates the entire class of SQL injection vulnerabilities across all queries.
server-v2.js: db.prepare('SELECT * FROM api_keys WHERE key = ?').get(key)
No eval() on user input outside sandbox
The application does not call eval(), new Function(), or any dynamic code evaluation on user-supplied strings outside the explicit VM sandbox. User code is only executed through the sandboxed exec-javascript API described above.

05

Monitoring & Audit Logging

Every authenticated API call produces an audit log record. Usage is queryable per key. An admin statistics endpoint provides server-wide visibility.

Per-call audit log in SQLite
Every API execution writes a row to the audit_log table containing: timestamp, key prefix (not the full key), API slug, credits consumed, latency in milliseconds, and the execution engine. Indexes on ts and api support efficient queries.
audit_log schema: ts, key_prefix, api, credits, latency_ms, engine
Per-key usage history
Authenticated users can query their own call history via GET /v1/usage, which returns the last 1,000 log entries for their key along with per-API totals and aggregate credit spend.
server-v2.js: dbGetAudit.all(keyPrefix) — last 1000 rows for the requesting key
Admin statistics endpoint
GET /v1/admin/stats (protected by ADMIN_SECRET) returns server-wide usage statistics: total calls, total credits consumed, top APIs by call volume, recent audit entries, and key count. This is used for operational monitoring.
Response headers expose per-call telemetry
Every API response includes X-Credits-Used, X-Credits-Remaining, X-Latency-Ms, X-Request-Id, and X-Engine headers. This allows clients to observe their own consumption without querying the usage endpoint.
server-v2.js: res.set('X-Credits-Used', ...) / res.set('X-Credits-Remaining', ...)

06

Infrastructure

The API server runs on Railway Pro. Static assets are served via Vercel's edge CDN. No multi-region failover is currently configured.

Railway Pro hosting with automatic TLS
The application is deployed on Railway's Pro tier, which provides automatic TLS certificate provisioning and renewal, DDoS mitigation at the network edge, and managed container restarts on failure.
Vercel edge CDN for static assets
The slopshop.gg frontend (HTML, CSS, JS) is deployed to Vercel's global edge network, served from locations close to the user. This reduces latency and provides edge-level DDoS protection for public pages.
WAL journaling for crash recovery
SQLite is configured in WAL mode. If the process is interrupted mid-write, WAL ensures the database returns to its last consistent state on next open. No data corruption occurs from unclean shutdowns.
server-v2.js: db.pragma('journal_mode = WAL')
Automatic rate-limit map cleanup
The in-memory IP rate-limit map is scanned every 5 minutes. Entries older than one hour are evicted. This prevents unbounded memory growth from a large number of unique IPs over time.
server-v2.js: setInterval(() => { /* evict entries > 1hr old */ }, 300000)
Multi-region redundancy
The API server runs in a single Railway region. There is no hot standby, read replica, or geographic failover. A regional Railway outage would cause downtime. See Security Roadmap.

07

Credit System Safety

Credit balance is checked before every execution. No API runs without sufficient balance. Deductions happen at the point of execution; the updated balance is persisted to SQLite immediately after.

Balance checked before execution — never execute on insufficient credits
Before calling any handler, the server verifies acct.balance >= def.credits. If the balance is insufficient and auto-reload is not configured, the request is rejected with HTTP 402 before any computation occurs.
server-v2.js: if (req.acct.balance < def.credits) return res.status(402)...
Atomic balance deduction with immediate persistence
The balance is decremented in memory before the handler runs, then persistKey() writes the new balance to SQLite via a prepared UPDATE statement after the response is built. The in-memory and on-disk balances are kept in sync on every call.
server-v2.js: req.acct.balance -= def.credits; ... persistKey(req.apiKey)
Auto-reload with configurable threshold
Users can configure auto-reload via POST /v1/credits/auto-reload, specifying a threshold and reload amount. When the balance drops below the threshold at the start of a call, the reload amount is added before execution proceeds. The configuration is stored in SQLite and survives server restarts.
server-v2.js: if (req.acct.auto_reload) { req.acct.balance += req.acct.auto_reload.amount; }
Key prefix only in audit log — full key never logged
The audit log stores only a prefix of the API key (key_prefix), not the full key. This means the audit log cannot be used to impersonate a user even if the log table were accessed directly.
server-v2.js: dbInsertAudit.run(ts, key.slice(0, 12), api, credits, latency, engine)

Security Roadmap

Upcoming security enhancements on the engineering roadmap.

Service Level

Uptime Target: 99.5%

Slopshop targets 99.5% uptime for the hosted API at slopshop.gg. The API runs on Railway Pro with automatic TLS, health monitoring, and zero-downtime deploys. Real-time status available at /status. Compute-tier APIs (189 endpoints) have zero external dependencies and are the most reliable. Network and LLM APIs depend on upstream providers.

For guaranteed SLAs with contractual commitments, contact dev@slopshop.gg for enterprise plans.

Engineering Roadmap

Scoped API keys with RBAC

Per-namespace, per-category, and read-only API key scoping. Team management with per-team credit budgets.

Process-level sandbox isolation

Replace vm.createContext with isolated-vm or container-level isolation for code execution APIs.

SOC 2 Type II certification

Formal compliance audit with independent assessor.

Encryption at rest

SQLite Encryption Extension for database-level encryption.

Multi-region redundancy

Geographic distribution with automated failover.
